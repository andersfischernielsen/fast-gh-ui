Architecture Overview

SSR SvelteKit SPA — SvelteKit in SSR mode for fast first paint, client-side navigation after hydration. Token stored in localStorage, Octokit calls from browser only. Static-ish deployment (adapter-auto works).

Component Tree

App
├── TokenGate          — login screen, token input
└── Shell              — authenticated layout
    ├── Sidebar        — repo list, filters
    └── MainPanel
        ├── NotificationsList    — chronological, filterable
        └── PRView
            ├── PRHeader         — title, status, meta
            ├── PRTabs           — Conversation | Commits | Checks | Files
            ├── Conversation     — comments threaded
            ├── Commits          — commit list with expand
            ├── Checks           — check runs + status
            └── FilesChanged     — file tree + diff viewer
                ├── FileTree     — collapsible file list
                ├── DiffViewer   — unified/split toggle, syntax highlight
                └── InlineComment — read/write comments on lines

Data Flow

  1. Octokit instantiated client-side with token from localStorage
  2. IndexedDB cache layer wraps Octokit calls — return cached data immediately, fetch+update in background
  3. GitHub API responses cached by URL + ETag for conditional requests
  4. Svelte 5 runes ($state, $derived, $effect) for reactive UI
  5. API calls happen in onMount or user-triggered actions, NOT in SvelteKit load functions (token not available server-side)
