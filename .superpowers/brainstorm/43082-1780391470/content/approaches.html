Architecture approaches

Three ways to structure the app, with trade-offs

Option A: SPA with client-side routing (Recommended)

Pros

  • Full control over navigation — instant transitions
  • Octokit runs entirely in browser (no backend needed)
  • IndexedDB caching integrates naturally
  • Simplest to deploy (static files)
  • Fastest UX — no server roundtrips for page loads

Cons

  • Larger initial JS bundle
  • All API tokens visible in browser storage

Option B: Hybrid — API proxy + SPA

Pros

  • Token stays server-side (more secure)
  • Can do server-side caching/aggregation

Cons

  • Requires a backend server to deploy/maintain
  • Token storage on server is a liability
  • More latency — every request goes through proxy
  • Defeats the "purely in browser" requirement

Option C: Multi-page app (MPA) with SvelteKit

Pros

  • Good initial load per page
  • SvelteKit SSR for fast first paint

Cons

  • Page transitions = full navigation
  • SSR doesn't help since Octokit calls happen on client
  • More complex data flow

Recommendation: Option A — pure SPA. Token-in-browser is the stated requirement, so the proxy's main advantage doesn't apply. SPA gives us the best performance (instant navigation, no server) and simplest deployment.
